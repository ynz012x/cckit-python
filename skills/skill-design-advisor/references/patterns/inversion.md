# Inversion 模式

## 定义

Inversion（反向/采访模式）让 Agent 先充分采访用户收集信息，确认所有信息完备后才执行任务。核心理念是"先问清楚，再动手" — 将通常由用户驱动的交互反转为由 Agent 驱动。

## 在 Claude Code 中的实现方式

### SKILL.md 结构建议

```markdown
---
name: <skill-name>
description: <描述 + 触发场景>
---

# <标题>

## 流程概述

1. 前置准备 — 收集核心参数
2. <信息挖掘步骤> — 通过交互完善信息
3. <执行步骤> — 基于收集的信息执行任务

---

## 前置准备

与用户确认以下信息：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `param_1` | ... | 必填 |
| `param_2` | ... | 可选 |

询问用户是否已有相关材料，如有则先阅读理解。

---

## 交互原则

**歧义澄清**：遇到以下情况必须暂停并向用户提问，不得自行假设：

| 歧义类型 | 说明 |
|----------|------|
| 语义不明 | 术语含义、规则不确定 |
| 边界模糊 | 范围、职责有多种解读 |
| 隐含需求 | 用户未提及但可能存在的需求 |

**澄清流程**：标记歧义 → 列出可能方案 → 给出推荐 → 等待回复

**阶段门禁**：每个阶段结束后须将产出展示给用户确认。

---

## Step 1: <信息收集阶段>

<具体的收集流程和要点>

## Step 2: <执行阶段>

<基于收集信息执行任务>
```

### 推荐目录布局

```
<skill-name>/
├── SKILL.md                       # 收集流程 + 执行流程
├── assets/
│   └── templates/                 # 输出模板（收集完后用于生成）
│       └── <output-template>.md
└── references/
    └── <methodology>.md           # 方法论参考（如需求分析框架、设计方法论）
```

### 关键设计要点

1. **明确的"前置准备"节**：用表格结构化要收集的参数，降低用户认知负担
2. **歧义澄清规则**：列出哪些情况必须暂停提问，避免 Agent 自行假设
3. **阶段门禁**：每阶段产出必须用户确认后才进入下一阶段
4. **控制交互轮次**：建议收集阶段控制在 3-5 轮，过长会导致注意力稀释

## 典型示例

**需求分析助手**（假设场景）：
- `前置准备` 节收集核心参数（项目名称、背景、分析深度、输出目录）
- 多阶段分析法，每阶段末尾有阶段门禁
- `交互原则` 节定义歧义类型（语义不明、边界模糊、隐含需求等）和澄清流程
- 分析深度可选（quick/standard/comprehensive），控制信息收集的深度和广度

核心结构特征：
```
前置准备（参数收集）→ 阶段1（问+分析）→ 确认 → 阶段2（问+分析）→ 确认 → 阶段3（问+分析）→ 确认 → 输出
```

**个性化推荐顾问**（假设场景）：
- 先通过 5-8 个问题了解用户偏好、预算、使用场景
- 收集完毕后一次性输出推荐方案
- 关键点：所有信息收集完毕前不输出任何推荐

## 适用场景

- 任务需求不明确，需要通过交互挖掘
- "先收集完所有信息，再一次性执行"的模式
- 执行前需要理解用户的上下文、约束和偏好
- 典型场景：需求分析、方案咨询、个性化配置

## 不适用场景

- 输入已经明确（不需要"采访"阶段）
- 需要边做边确认（那是 Pipeline，不是 Inversion）
- 全自动执行的任务（用 Tool Wrapper 或 Generator）

## 常见陷阱

1. **注意力稀释**：5-10 轮交互后模型可能遗忘早期信息或跳过阶段门禁
   - 缓解：每轮结束输出结构化摘要，每阶段开头"回顾已收集信息"
2. **阶段门禁无框架保障**：Claude Code 中门禁全靠自然语言约束
   - 缓解：在每步开头加"确认上一步已获用户批准"
3. **收集阶段过长**：超过 5 轮交互时考虑是否应该拆分
4. **收集和执行界限模糊**：应明确标记"以下为执行阶段，不再收集新信息"

## 与其他模式的组合

- **Inversion + Generator**：最常见组合。先采访收集信息，再按模板生成输出
- **Inversion + Pipeline**：信息收集后进入多步执行流程
- **Inversion 作为 Pipeline 的 Step 1**：Pipeline 首步用 Inversion 收集需求，后续步骤逐步执行
