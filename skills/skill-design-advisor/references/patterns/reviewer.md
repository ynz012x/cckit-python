# Reviewer 模式

## 定义

Reviewer 用外部检查清单对内容进行结构化评审。skill 的核心价值在于"按标准逐项检查，给出客观评估"，检查清单定义了评审维度和判定标准。

## 在 Claude Code 中的实现方式

### SKILL.md 结构建议

```markdown
---
name: <skill-name>
description: <描述 + 触发场景>
---

# <标题>

## 流程概述

1. 获取待审内容
2. 加载检查清单
3. 逐项评审
4. 输出评审报告

---

## Step 1: 获取待审内容

<说明如何获取输入：读取文件/接收用户粘贴/分析代码>

## Step 2: 逐项评审

加载检查清单 [assets/checklists/<checklist>.md](assets/checklists/<checklist>.md)，对每个检查项进行评估。

评估输出格式：

| 检查项 | 状态 | 说明 |
|--------|------|------|
| <项目名称> | 通过/不通过/不适用 | <具体说明> |

## Step 3: 输出评审报告

汇总评审结果：
- 总体评级
- 通过项数 / 总项数
- 不通过项的改进建议
```

### 推荐目录布局

```
<skill-name>/
├── SKILL.md                    # 评审流程
└── assets/
    └── checklists/
        └── <checklist>.md      # 检查清单
```

如果评审需要参考外部规范：
```
<skill-name>/
├── SKILL.md
├── assets/checklists/
│   └── <checklist>.md
└── references/
    └── <standard>.md           # 评审参考的规范文档
```

### 关键设计要点

1. **检查项必须可判定**：每项应该能明确判断"通过/不通过"，避免主观模糊项
2. **检查清单独立于 SKILL.md**：放在 assets/checklists/ 中，便于维护和更新
3. **评审输出要结构化**：表格形式展示逐项结果，而非散文描述
4. **区分严重级别**：可将检查项分为 Error / Warning / Info 三级

## 典型示例

**独立 Reviewer Skill**（假设场景 — PR 代码审查）：
- SKILL.md 定义评审流程：获取 diff → 加载检查清单 → 逐项评估 → 输出报告
- `assets/checklists/code-review.md` 包含 20+ 检查项（命名规范、错误处理、安全性等）
- 输出结构化表格：每项 通过/不通过 + 具体位置和改进建议

**作为子步骤嵌入 Pipeline**（常见用法）：
- 在多步骤流程的每个阶段末尾，用 checklist 做完整性验证
- 每个阶段有独立的检查清单文件：`assets/checklists/phase1-checklist.md` 等
- 检查结果作为阶段门禁 — 全部通过才进入下一阶段
- 这是 Reviewer 模式最常见的使用方式：不作为独立 skill，而是嵌入其他模式中

## 适用场景

- 输出是评估结果、检查报告、合规判定
- 有明确的检查标准或评审维度
- 核心工作是"对照标准逐项检查"
- 典型场景：代码审查、安全合规检查、文档质量评估、架构评审

## 不适用场景

- 没有明确的检查标准（无清单就不是 Reviewer）
- 输出是生成内容而非评估内容（用 Generator）
- 评审标准高度主观，无法用通过/不通过判定

## 常见陷阱

1. **检查项过于宽泛**："代码质量好"不是可判定的检查项，应拆为具体子项
2. **只有 SKILL.md 没有独立 checklist 文件**：检查清单应独立维护，方便复用和更新
3. **评审输出不结构化**：散文式的评审意见难以追踪和复核，应用表格呈现
4. **强制对主观任务使用评分**：写作风格、设计美感等主观维度不适合 Reviewer 的通过/不通过模式

## 与其他模式的组合

- **Pipeline + Reviewer**：最自然的组合。Pipeline 每步结束后用 Reviewer 做阶段门禁检查
- **Generator + Reviewer**：生成后验证。先按模板生成，再用检查清单验证完整性
- **Tool Wrapper + Reviewer**：加载外部规范作为评审参考标准
