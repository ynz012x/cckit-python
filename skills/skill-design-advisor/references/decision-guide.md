# 模式选择指南

本文档提供 5 种设计模式的选择方法，基于多维度信号分析而非单一决策树。

## 分析维度与信号

### 维度1: 输出形态

| 输出类型 | 强烈指向 | 可能指向 |
|----------|---------|---------|
| 生成文件（代码/文档/配置） | Generator | Pipeline（如果生成过程分多步） |
| 检查报告/评估结果/打分 | Reviewer | Generator（如果报告有固定模板） |
| 分析建议/方案推荐 | Inversion（需收集信息） | Tool Wrapper（有领域知识参考） |
| 执行操作（git commit/部署/安装） | Tool Wrapper | Pipeline（如果操作有多步） |

### 维度2: 交互复杂度

| 交互轮次 | 指向 | 说明 |
|----------|------|------|
| 0（全自动） | Tool Wrapper / Generator | 输入明确，无需交互 |
| 1-2 轮 | Generator / Reviewer | 简单确认即可执行 |
| 3-5 轮 | Inversion | 需要挖掘需求 |
| 5+ 轮 | Pipeline | 多阶段交互确认 |

### 维度3: 领域知识需求

| 知识需求 | 指向 | 典型场景 |
|----------|------|---------|
| 需要加载规范/API文档 | Tool Wrapper | 按编码规范检查、按 API 文档生成 |
| 需要检查清单 | Reviewer | 合规检查、代码审查 |
| 需要方法论框架 | Pipeline / Inversion | 需求分析、系统设计 |
| 不需要外部知识 | Generator | 模板填充 |

### 维度4: 流程结构

| 流程特征 | 指向 | 说明 |
|----------|------|------|
| 线性执行，无需确认 | Tool Wrapper / Generator | 一步到位 |
| 分步执行，步间需确认 | Pipeline | 严格顺序 + 阶段门禁 |
| 先收集后执行 | Inversion | 前半段交互，后半段执行 |
| 对已有内容做评审 | Reviewer | 输入是待审内容 |

## 综合分析方法

对每种模式评估 4 个维度的信号强度，形成分析表：

```

┌─────────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│ 维度         │ Tool Wr. │ Generator│ Reviewer │Inversion │ Pipeline │
├─────────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│ 输出形态     │  ○       │  ●       │  ○       │  ○       │  ○       │
│ 交互复杂度   │  ●       │  ●       │  ○       │  ○       │  ○       │
│ 领域知识     │  ●       │  ○       │  ○       │  ○       │  ○       │

```

## 边界案例处理

### 案例1: Generator vs Pipeline
**场景**："生成文档，但需要先收集 3 个信息"
- 如果收集信息只是简单参数（名称、路径），用 **Generator** + 前置准备
- 如果收集信息需要多轮讨论和确认，用 **Pipeline**

### 案例2: Reviewer vs Tool Wrapper
**场景**："按规范检查代码"
- 如果输出是结构化的检查报告（通过/不通过），用 **Reviewer**
- 如果只是参考规范来辅助其他操作，用 **Tool Wrapper**

### 案例3: Inversion vs Pipeline
**场景**："需要多轮交互的复杂任务"
- **Inversion**：先问完所有问题，再一次性执行（"采访 → 执行"两阶段）
- **Pipeline**：交替执行和确认，每步都有产出（"做一步 → 确认 → 做下一步"）

### 案例4: 多模式同等匹配
当两种及以上模式信号强度接近时：
1. 列出所有候选模式的 pros/cons
2. 优先推荐更简单的模式（Tool Wrapper > Generator > Reviewer > Inversion > Pipeline）
3. 考虑将强信号模式作为"主模式"，次强信号模式作为"辅助模式"

### 案例5: 需求超出单一 skill 范围
当分析发现 skill 涉及多个独立职责时，建议拆分为多个 skill，并说明各 skill 之间的调用关系。判断标准：如果 Pipeline 超过 7 步，或涉及 3 个以上完全不同的领域知识。

## 模式组合规则

| 组合 | 触发条件 | 推荐实现 |
|------|---------|---------|
| Pipeline + Reviewer | Pipeline 的某些步骤需要验证产出 | 添加 `assets/checklists/`，在关键步骤设完整性检查 |
| Inversion + Generator | 收集完信息后需要按模板输出 | 添加 `assets/templates/`，收集阶段后切入模板生成 |
| Generator + Reviewer | 生成内容后需要质量验证 | 在生成步骤后添加独立的"完整性检查"步骤 |
| Tool Wrapper + 任意 | 任何模式中需要加载领域知识 | 添加 `references/` 目录存放参考文档 |
| Pipeline + Inversion | Pipeline 的首步需要信息挖掘 | 将 Inversion 作为 Pipeline 的 Step 1 |

## 反模式警告

1. **单步骤用 Pipeline**：如果 skill 只有 1-2 步，不需要 Pipeline 的复杂性
2. **无检查清单用 Reviewer**：Reviewer 的核心是检查清单，没有清单就不是 Reviewer
3. **全自动用 Inversion**：如果输入已经明确，不需要"采访"阶段
4. **无模板用 Generator**：如果输出格式不固定，Generator 模式价值有限
5. **所有内容塞进 SKILL.md**：超过 500 行应拆分到 references/
